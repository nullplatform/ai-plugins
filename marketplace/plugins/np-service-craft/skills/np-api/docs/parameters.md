# Parameters, Values, and Versions

Environment variables and hierarchical configuration. Parameters have a two-level structure:
- **Parameter**: Definition (name, type, secret, NRN)
- **Parameter Values**: Concrete values per dimension (e.g., one value for production, another for development)
- **Parameter Versions**: Change history of the parameter

## @endpoint /parameter

Lists parameters by NRN.

### Parameters

- `nrn` (query, required): Hierarchical NRN

### NRN Formats

```
organization={org_id}:account={acc_id}:namespace={ns_id}:application={app_id}
organization={org_id}:account={acc_id}:namespace={ns_id}:application={app_id}:scope={scope_id}
```

### Response

```json
{
  "paging": {"offset": 0, "limit": 30, "total": 5},
  "results": [
    {
      "id": 671080587,
      "name": "CONSULTING_PRICES_DB_HOSTNAME",
      "nrn": "organization=...:account=...:namespace=...:application=...",
      "type": "environment",
      "encoding": "plaintext",
      "variable": "CONSULTING_PRICES_DB_HOSTNAME",
      "secret": false,
      "read_only": true,
      "version_id": 1791955022,
      "values": [
        {
          "id": "683303163",
          "nrn": "organization=...:account=...:namespace=...:application=...",
          "value": "172.20.144.91",
          "created_at": "2026-02-12T14:44:02.872Z",
          "dimensions": {"environment": "production"},
          "external": null
        }
      ]
    }
  ]
}
```

### Parameter fields

- `id`: Numeric ID
- `name`: Parameter name (UPPER_SNAKE_CASE)
- `nrn`: NRN of the level where it was defined (application or scope)
- `type`: `environment` (environment variable) | `linked_service` (generated by service link)
- `encoding`: `plaintext`
- `variable`: Name of the env var injected into the container (generally same as `name`)
- `secret`: boolean - whether the value is masked in UI and logs
- `read_only`: boolean - `true` for parameters generated by service links
- `version_id`: ID of the current parameter version
- `values[]`: Array of values per dimension (see structure below)

### Value fields

- `id`: Value ID (string)
- `nrn`: NRN where the value applies
- `value`: Concrete value (string). `null` if `secret: true`
- `dimensions`: Object with the dimensions where it applies (e.g., `{"environment": "production"}`)
- `created_at`: Creation timestamp
- `external`: External reference (null if not applicable)

### Notes

- A parameter can have multiple values for different dimensions (e.g., one value for production, another for development)
- When querying with scope NRN, returns resolved parameters for that scope (application + scope-level)
- Parameters with `secret: true` return `value: null` on read
- Parameters with `read_only: true` were generated by service links and cannot be modified
- The actual `type` in the API is `environment`, not `variable`

### Example

```bash
# Application parameters
np-api fetch-api "/parameter?nrn=organization=1255165411:account=95118862:namespace=463208973:application=2052735708"

# Resolved parameters for a specific scope
np-api fetch-api "/parameter?nrn=organization=1255165411:account=95118862:namespace=463208973:application=2052735708:scope=675195311"
```

## @endpoint /parameter/{id}

Gets an individual parameter by ID.

### Parameters

- `id` (path, required): Parameter ID

### Response

Same structure as each element of `/parameter`, including `values[]`.

### Navigation

- **→ values**: Embedded in the response as `values[]`
- **→ versions**: `/parameter/{id}/version`
- **← application**: `/parameter?nrn={application_nrn}`

### Example

```bash
np-api fetch-api "/parameter/671080587"
```

---

## @endpoint /parameter/{id}/version

Lists all available versions of a parameter.

### Parameters

- `id` (path, required): Parameter ID

### Response

```json
{
  "results": [
    {
      "id": 1791955022,
      "created_at": "2026-02-12T14:44:02.813Z",
      "user_id": 629868107
    }
  ]
}
```

### Fields

- `id`: Version ID
- `created_at`: Creation timestamp
- `user_id`: ID of the user who created the version

### Navigation

- **→ user**: `user_id` → `/user/{user_id}`

### Example

```bash
np-api fetch-api "/parameter/671080587/version"
```

## @endpoint /parameter/{parameterId}/value/{sourceValueId}/compare/{targetValueId}

Compares two values of the same parameter.

### Parameters

- `parameterId` (path, required): Parameter ID
- `sourceValueId` (path, required): First value ID
- `targetValueId` (path, required): Second value ID

### Example

```bash
np-api fetch-api "/parameter/671080587/value/683303163/compare/683303164"
```

### Notes

- Useful for comparing values between dimensions (e.g., production vs development)

## @endpoint /nrn/{nrn}

Gets platform-level configuration by NRN.

### Parameters

- `nrn` (path, required): Hierarchical NRN (URL encoded)
- `ids` (query, required): List of keys to get (comma-separated)

### ID Format

```
namespace.key
```

Common namespaces: `global`, `prod`, `staging`

### Response

```json
{
  "nrn": "organization=123:account=456",
  "namespaces": {
    "global": {
      "key1": "value1"
    },
    "prod": {
      "key2": "value2"
    }
  },
  "omittedKeys": ["nonexistent.key"]
}
```

### Example

```bash
np-api fetch-api "/nrn/organization=549683990:account=463975847:namespace=476951634?ids=global.key1,prod.key2"
```

### Notes

- Requires explicit `ids` - does not return all keys
- Non-existent keys appear in `omittedKeys`
- Different from `/parameter` - this is internal platform config
