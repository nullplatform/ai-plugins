# Nullplatform Bindings Layer Generation Guide

Generation guide for the **nullplatform-bindings/** layer files in Nullplatform with OpenTofu.

> **IMPORTANT**: Do not assume values or configurations. When in doubt, divergence, or ambiguity, **always ask the user** before generating code. It's better to ask additional questions than to generate incorrect code or with wrong assumptions.

> **CONTINUOUS IMPROVEMENT**: After each generation, if errors, divergences, or improvements are found, **automatically add them to the "Lessons learned" section** of this PROMPT to build standardized knowledge.

> **MANDATORY VALIDATION**: Before generating or modifying any code, **validate that it meets ALL the rules** of this document. Validation checklist:
> 1. Did I read the downloaded module's `variables.tf` to know the required variables? (Technical rule #7)
> 2. Am I only including variables without default (mandatory)?
> 3. Am I inferring any value that I should ask the user about?
> 4. Am I using the module instead of creating resources directly?
> 5. Are the module blocks clean, without inline comments? (Rule #6)
> 6. Are the module blocks in main.tf in alphabetical order by name?
> 7. Do infrastructure values use `local.*` with fallback? Do nullplatform values use `local.*` from remote state? (data.tf/locals.tf pattern)
> 8. Are there no scope/service specification variables in variables.tf or terraform.tfvars?

> **EXECUTION INSTRUCTIONS**: Mandatory flow for generating correct code:
> 1. Follow the question flow using `AskUserQuestion` - DO NOT assume configurations
> 2. After generating, run `tofu init -backend=false && tofu validate`
> 3. To verify module variables, read from `.terraform/modules/<name>/variables.tf` (downloaded version), **NEVER** from the repository root
> 4. If validate fails, fix BEFORE considering it done
> 5. **DO NOT make changes** to already generated code without explicit user confirmation
> 6. When reverting changes, verify with `grep -r` that no residual references remain

## What modules go in nullplatform-bindings/

The `nullplatform-bindings/` folder connects Nullplatform with cloud and code repository. It contains the bindings between the platform and external services.

### Available modules

```hcl
# Code repository
module "code_repository" { }  # nullplatform/code_repository

# Asset repository (by cloud)
module "asset_repository" { }  # nullplatform/asset/ecr (AWS) or nullplatform/asset/docker_server

# Cloud provider config
module "cloud_provider" { }  # nullplatform/cloud/aws/cloud (or azure/gcp)

# Scope-agent associations
module "scope_definition_channel_association" { }  # nullplatform/scope_definition_agent_association

# Service definition associations (optional)
module "service_definition_channel_association" { }  # nullplatform/service_definition_agent_association

# Monitoring
module "monitoring_provider" { }  # nullplatform/metrics

# API Keys for notifications
module "scope_notification_api_key" { }    # nullplatform/api_key
module "service_notification_api_key" { }  # nullplatform/api_key (optional, if there are service definitions)
```

## Question flow

### Step 1: Components
- **Which code repository?**: github | gitlab | azure
- **Asset repository**: ECR (AWS) | Docker Server (any cloud)
- **Cloud provider**: Determined automatically based on the cloud chosen in infrastructure

### Step 2: Code repository specific variables

After choosing the code repository, request the required variables. These are determined by reading the downloaded module's `variables.tf` (`code_repository`), filtering Level 2 variables that apply to the chosen provider (they have conditional `validation` with `var.git_provider != "provider"`).

**IMPORTANT**: Do not hardcode the variable list here. Always read from the downloaded module. As a guide:

- **GitHub**: Variables starting with `github_`
- **GitLab**: Variables starting with `gitlab_` (note: some are sensitive, others are complex objects)
- **Azure**: Variables starting with `azure_` (check which ones have defaults)

For each required variable of the chosen provider:
1. Declare it in bindings' `variables.tf` (with `sensitive = true` if applicable)
2. If the variable has a `default` in the module, offer that default as an option to the user (often the default is sufficient)
3. If the variable does NOT have a default or the user wants to change the default, request the value

### Step 3: Summary before generating

Show a table with all components and captured variables. Wait for confirmation before generating.

## Generation rules

### IMPORTANT: Do not assume, always ask

1. **NEVER assume variable values** - If you don't have explicit information from the user, ask
2. **When in doubt or ambiguity, ask**
3. **If there are divergences between what the user says and what you see in the modules, ask**
4. **If a module requires variables that weren't mentioned, ask**

### Technical rules

1. **Use the latest published version** in all modules
   - **Before generating**, run `git tag --sort=-v:refname | head -1` to get the latest release

2. **Required providers**:
   - AWS: `hashicorp/aws ~> 6.0`
   - `nullplatform/nullplatform` - Check the latest version before generating:
     ```bash
     curl -s "https://registry.terraform.io/v1/providers/nullplatform/nullplatform/versions" | jq -r '[.versions[].version] | sort_by(split(".") | map(tonumber)) | last'
     ```
     Use `~> 0.0.X` with the obtained version. DO NOT hardcode a fixed version.

3. **Sensitive variables** marked correctly

4. **Format with `terraform fmt`** after generating

5. **Read module READMEs** before generating

6. **Alphabetical order in main.tf** - The `module` blocks in main.tf must be sorted alphabetically by name

7. **NEVER transform outputs between modules** - Pass outputs as-is (without `replace`, `regex`, `split`, etc.). If in doubt about the format a variable expects, read the module's internal code (main.tf, iam.tf, locals.tf) to see how it's used, don't infer by variable name

8. **MANDATORY: Read variables.tf of EACH module before generating main.tf** - After `tofu init -backend=false`, read the downloaded `variables.tf` of each module in `.terraform/modules/<name>/`. Only include variables without default (mandatory) and only then generate the module blocks. Don't trust the patterns in this document - they may be outdated compared to the module version.

9. **Generate one api_key per scope defined in nullplatform** - For each scope that exists (containers, scheduled_task, etc.), create an `api_key` module with `type = "scope_notification"` and a distinctive name. Then pass that api_key to the corresponding channel association:
   ```hcl
   module "scope_notification_api_key" {
     source             = "...//nullplatform/api_key?ref=vX.Y.Z"
     type               = "scope_notification"
     nrn                = var.nrn
     specification_slug = local.scope_specification_slug
   }

   module "scope_notification_api_key_scheduled_task" {
     source             = "...//nullplatform/api_key?ref=vX.Y.Z"
     type               = "scope_notification"
     nrn                = var.nrn
     specification_slug = local.scope_specification_slug_scheduled_task
   }

   module "scope_definition_channel_association" {
     api_key = module.scope_notification_api_key.api_key
   }

   module "scope_definition_channel_association_scheduled_task" {
     api_key = module.scope_notification_api_key_scheduled_task.api_key
   }
   ```
   The same applies for service_definitions if they exist (with `type = "service_notification"`).

## tfvars rules

### Separation of common and specific variables

1. **`common.tfvars`** (in root): Shared variables
   - `np_api_key`, `nrn`, `tags_selectors`

2. **`terraform.tfvars`** (in nullplatform-bindings/): Only specific variables
   - Code repository variables (github_*, gitlab_*, azure_*)
   - DO NOT duplicate variables that are in common.tfvars

### Usage pattern

```hcl
#
# Nullplatform Bindings - Specific Variables
#
# Usage: tofu plan -var-file=../common.tfvars -var-file=./terraform.tfvars
#
```

## data.tf and locals.tf pattern (Remote State + Locals Fallback)

The `nullplatform-bindings/` layer consumes outputs from previous layers via `terraform_remote_state`. This is the only way to get scope/service definition values - **they are NEVER declared as variables nor hardcoded in tfvars**.

### Rule: Value classification by source

| Source | Pattern | Example |
|--------|---------|---------|
| **Infrastructure** | Conditional fallback: variable `default = null` + ternary | `cluster_name`, `domain_name`, zone IDs |
| **Nullplatform** | Always from remote state, no fallback or variable | `scope_specification_id`, `scope_specification_slug` |
| **Bindings-specific** | Direct variable in `variables.tf` + tfvars | `github_organization`, `git_provider` |

### 1. data.tf - Remote state references

Two `terraform_remote_state` blocks:

**Infrastructure (conditional with count):**
```hcl
data "terraform_remote_state" "infrastructure" {
  count   = (var.cluster_name == null || var.domain_name == null || ...) ? 1 : 0
  backend = "local"  # or "s3", depending on infrastructure's backend.tf
  config = {
    path = "../infrastructure/{cloud}/terraform.tfstate"
  }
}
```
- The `count` evaluates if ANY infrastructure variable is `null`. If all were provided via tfvars, the remote state is not read.
- The count condition must include ALL infrastructure variables that are consumed.

**Nullplatform (always, without count):**
```hcl
data "terraform_remote_state" "nullplatform" {
  backend = "local"  # or "s3", depending on nullplatform's backend.tf
  config = {
    path = "../nullplatform/terraform.tfstate"
  }
}
```
- Without `count` because it's a hard dependency. Scope/service IDs and slugs always come from here.

### 2. locals.tf - Value resolution

```hcl
locals {
  # Infrastructure - variable overrides remote state
  cluster_name    = var.cluster_name != null ? var.cluster_name : data.terraform_remote_state.infrastructure[0].outputs.cluster_name
  domain_name     = var.domain_name != null ? var.domain_name : data.terraform_remote_state.infrastructure[0].outputs.domain_name

  # Nullplatform - always from remote state
  scope_specification_id   = data.terraform_remote_state.nullplatform.outputs.scope_specification_id
  scope_specification_slug = data.terraform_remote_state.nullplatform.outputs.scope_specification_slug
}
```

**locals.tf rules:**
- Infrastructure: ternary `var.x != null ? var.x : data.terraform_remote_state.infrastructure[0].outputs.x`
- Nullplatform: direct assignment `data.terraform_remote_state.nullplatform.outputs.x`
- Local names must match the output names of each layer

### 3. variables.tf - Only infrastructure with default = null

Infrastructure variables are declared with `default = null` to allow override:

```hcl
variable "cluster_name" {
  type    = string
  default = null
}
```

**DO NOT declare variables for nullplatform values** (scope_specification_id, scope_specification_slug, etc.). These only exist in locals.tf via remote state.

### 4. main.tf - Always use local.*

Modules reference `local.*` for infrastructure and nullplatform values:

```hcl
module "cloud_provider" {
  domain_name = local.domain_name        # NOT var.domain_name
}

module "scope_definition_channel_association" {
  scope_specification_id = local.scope_specification_id  # NOT var.scope_specification_id
}
```

The only variables passed directly with `var.*` are bindings-specific ones (code repo, np_api_key, nrn, tags_selectors).

### 5. terraform.tfvars - No scope/service values

Scope/service specification IDs and slugs **NEVER go in terraform.tfvars**. Infrastructure values are optional (override).

### Backend config

Remote state connection data (bucket, key, region, profile) is read from each layer's `backend.tf` (`infrastructure/backend.tf` and `nullplatform/backend.tf`). DO NOT invent these values.

### Consumed outputs

**From infrastructure/outputs.tf:**

| Local variable | Description | Where used |
|---------------|-------------|------------|
| `cluster_name` | K8s cluster name | asset_repository |
| `domain_name` | Application domain | cloud_provider |
| `public_zone_id` | Public DNS zone ID | cloud_provider |
| `private_zone_id` | Private DNS zone ID | cloud_provider |

**From nullplatform/outputs.tf:**

For each scope definition: 2 outputs (id + slug). For each service definition: 2 outputs (id + slug). Exact names are determined by reading `nullplatform/outputs.tf`.

## Lessons learned

### 1. DO NOT use depends_on with ECR module
The `nullplatform/asset/ecr` module has local provider configurations. DO NOT use `depends_on`.

### 2. Output mapping between scope_definition and scope_definition_agent_association
Modules use different names: `service_specification_id` -> `scope_specification_id`, `service_slug` -> `scope_specification_slug`. The mapping is resolved in `nullplatform/outputs.tf`.

### 3. `np_api_key` vs `api_key` - they are different things
- `np_api_key`: Nullplatform provider authentication. Used by modules that only need auth (code_repository, asset, cloud, metrics).
- `api_key`: Generated at runtime by the `nullplatform/api_key` module. Used by `scope_definition_agent_association` and `service_definition_agent_association`. Available types: agent, scope_notification, service_notification, custom.

## Variable rules

### 5. How to determine if a variable is required

When reading a module's `variables.tf`, apply a simple rule:

- **Without `default`** → MANDATORY. Always include in main.tf, ask the user for the value.
- **With `default`** → DO NOT include in main.tf, DO NOT ask the user. Only include if this document explicitly indicates it or if the user requests it.

### Module reading flow

1. **Read `variables.tf`** of the downloaded module (`.terraform/modules/<name>/variables.tf`)
2. **Identify variables without default** → always include, ask the user for the value
3. **Variables with default** → DO NOT include, DO NOT ask

---

### 6. Do not add inline comments inside module blocks

Module blocks must be clean, without inline comments. Only use separation comments BEFORE the module block.

---

### 6b. Order within each module block

`source` goes first, then variables sorted alphabetically, and `depends_on` goes last separated by a blank line from the rest:

```hcl
module "example" {
  source = "git::https://..."

  alpha_var = "a"
  beta_var  = "b"
  zeta_var  = "z"

  depends_on = [module.other]
}
```

---

### 7. NEVER hardcode values - always use variables

- **Rule**: Every value in a module must come from a variable with its corresponding declaration in `variables.tf`

### 8. Consistent variable structure

For each variable used in main.tf:
1. Declare it in `variables.tf` with type, description, and default (if applicable)
2. If shared between layers, verify it's in `common.tfvars`

---

### 9. tags_selectors is a common variable

The `tags_selectors` variable is used in:
- `infrastructure/` -> agent module (`tags_selectors`)
- `nullplatform-bindings/` -> scope_definition_agent_association module (`tags_selectors`)

Since they must always have the **same values**, this variable should be in `common.tfvars`.

**In nullplatform-bindings/variables.tf:**
```hcl
variable "tags_selectors" {
  type        = map(string)
  description = "Tags selectors for agent channel filtering"
}
```

### 10. Use OpenTofu (tofu) instead of Terraform

IaC commands must use `tofu` instead of `terraform`.

---

### 11. Layer-specific variables go in their terraform.tfvars, not in common.tfvars

- Only put in `common.tfvars` variables that are used in **more than one** layer:
  - `common.tfvars`: np_api_key, nrn, tags_selectors, aws_region, domain_name
  - `nullplatform-bindings/terraform.tfvars`: gitlab_*, github_*, azure_* (code repository config)
