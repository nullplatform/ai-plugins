# Nullplatform Layer Generation Guide

Generation guide for the **nullplatform/** layer files in Nullplatform with OpenTofu.

> **IMPORTANT**: Do not assume values or configurations. When in doubt, divergence, or ambiguity, **always ask the user** before generating code. It's better to ask additional questions than to generate incorrect code or with wrong assumptions.

> **CONTINUOUS IMPROVEMENT**: After each generation, if errors, divergences, or improvements are found, **automatically add them to the "Lessons learned" section** of this PROMPT to build standardized knowledge.

> **MANDATORY VALIDATION**: Before generating or modifying any code, **validate that it meets ALL the rules** of this document. Validation checklist:
> 1. Did I read the module's `variables.tf` to know the mandatory variables (without default)?
> 2. Am I including ONLY variables without default in the module blocks of main.tf?
> 3. Am I inferring any value that I should ask the user about?
> 4. Am I using the module instead of creating resources directly?
> 5. Are the module variables correct for the current version?
> 6. Are the module blocks clean, without inline comments?
> 7. If I'm reusing a module, am I passing the variables that distinguish each instance? (Rule #16)

> **EXECUTION INSTRUCTIONS**: Mandatory flow for generating correct code:
> 1. Follow the question flow using `AskUserQuestion` - DO NOT assume configurations
> 2. After generating, run `tofu init -backend=false && tofu validate`
> 3. To verify module variables, read from `.terraform/modules/<name>/variables.tf` (downloaded version), **NEVER** from the repository root
> 4. If validate fails, fix BEFORE considering it done
> 5. **DO NOT make changes** to already generated code without explicit user confirmation
> 6. When reverting changes, verify with `grep -r` that no residual references remain

## What modules go in nullplatform/

The `nullplatform/` folder contains **only central Nullplatform configuration** at the organization level. It does NOT contain cloud modules, code repository, or bindings.

### Available modules

```hcl
# Scope definitions (at organization level)
module "scope_definition" { }                # nullplatform/scope_definition - ALWAYS included
module "scope_definition_scheduled_task" { } # nullplatform/scope_definition - OPTIONAL

# Dimensions
module "dimensions" { }  # nullplatform/dimensions

# Service definitions (optional)
module "service_definition_endpoint_exposer" { }  # nullplatform/service_definition - OPTIONAL
```

### What does NOT go in nullplatform/

The following modules go in `nullplatform-bindings/`, NOT here:
- `code_repository` - connection with GitLab/GitHub
- `asset_repository` - ECR or Docker Server
- `cloud_provider` - cloud configuration (AWS/Azure/GCP)
- `scope_definition_agent_association` - association with agents
- `metrics` - monitoring

## Generation rules

### IMPORTANT: Do not assume, always ask

1. **NEVER assume variable values** - If you don't have explicit information from the user, ask
2. **When in doubt or ambiguity, ask** - It's better to ask too much than generate incorrect code
3. **If there are divergences between what the user says and what you see in the modules, ask** for clarification
4. **If a module requires variables that weren't mentioned, ask** if they should be included
5. **If you're not sure which components to include, ask** - don't assume the "recommended" configuration

### Technical rules

1. **Use the latest published version** in all modules
   - **Before generating**, run `git tag --sort=-v:refname | head -1` to get the latest release from the repository
   - Use that version in all `?ref=vX.Y.Z` of the modules

2. **Required providers** - Check the latest nullplatform provider version before generating:
   ```bash
   curl -s "https://registry.terraform.io/v1/providers/nullplatform/nullplatform/versions" | jq -r '[.versions[].version] | sort_by(split(".") | map(tonumber)) | last'
   ```
   - Use `~> 0.0.X` with the obtained version (e.g., if the latest is `0.0.77`, use `~> 0.0.77`)
   - DO NOT hardcode a fixed version in this document

3. **Sensitive variables** marked correctly

4. **Format with `terraform fmt`** after generating

5. **Read module READMEs** before generating to verify current variables and dependencies

6. **NEVER transform outputs between modules** - Pass outputs as-is (without `replace`, `regex`, `split`, etc.). If in doubt about the format a variable expects, read the module's internal code (main.tf, iam.tf, locals.tf) to see how it's used, don't infer by variable name

7. **Alphabetical order in main.tf** - The `module` blocks in main.tf must be sorted alphabetically by name

8. **Order within each module block** - `source` goes first, then variables sorted alphabetically, and `depends_on` goes last separated by a blank line from the rest:
    ```hcl
    module "example" {
      source = "git::https://..."

      alpha_var = "a"
      beta_var  = "b"
      zeta_var  = "z"

      depends_on = [module.other]
    }
    ```

7. **Verify action_spec_names against the scopes repo** - DO NOT trust the module default or the patterns in this document. Before generating, check the actual available actions at `github.com/nullplatform/scopes`:
   ```bash
   gh api repos/nullplatform/scopes/contents/{service_path}/specs/actions --jq '.[].name'
   ```
   - The `{service_path}` corresponds to the scope (e.g., `k8s`, `scheduled_task`)
   - Use the complete list from the repo as the value for `action_spec_names`
   - The default in the `tofu-modules` module may be outdated compared to the scopes repo

## tfvars rules

### Separation of common and specific variables

1. **`common.tfvars`** (in root): Variables shared across all modules
   - `np_api_key`, `nrn`

2. **`terraform.tfvars`** (in nullplatform/): Only layer-specific variables
   - `service_path`, `service_path_scheduled_task`, `environments`
   - DO NOT duplicate variables that are in common.tfvars

### Usage pattern

```hcl
#
# Nullplatform - Specific Variables
#
# Usage: tofu plan -var-file=../common.tfvars -var-file=./terraform.tfvars
#
```

## Mandatory outputs for nullplatform-bindings consumption

The `nullplatform/` layer MUST export outputs in `outputs.tf` so that `nullplatform-bindings/` can consume them via `terraform_remote_state`.

### Rule: 2 outputs per scope definition, 2 outputs per service definition

For each `scope_definition` included, generate **2 outputs**:
- `scope_specification_id` (or with suffix if there's more than one: `_scheduled_task`)
- `scope_specification_slug` (same suffix)

For each `service_definition` included, generate **2 outputs**:
- `service_specification_id_{name}` (e.g., `_endpoint_exposer`)
- `service_specification_slug_{name}`

Each output takes its value from the module that created that scope or service in the generated `main.tf`. Read the downloaded module's `outputs.tf` in `.terraform/modules/` to know the real output names and map them to the contract names above.

**IMPORTANT**: These output names are a contract with `nullplatform-bindings/`. If they change, bindings will break.

## Lessons learned

### 1. nullplatform/ must NOT contain cloud/repository modules
- **Error**: Putting code_repository, asset_repository (ECR), or cloud_provider in the nullplatform/ folder
- **Solution**: The `nullplatform/` folder should only contain:
  - `scope_definition` - organization-level scope definitions
  - `dimensions` - environment configuration
  - `service_definition` - service definitions (optional)
- **The following modules go in nullplatform-bindings/**:
  - `code_repository`, `asset_repository`, `cloud_provider`

### 2. nullplatform/ folder is simpler and doesn't require infrastructure data sources
- **Error**: Including infrastructure data sources (cluster, VPC, DNS zones) in nullplatform/
- **Solution**: The `nullplatform/` modules (scope_definition, dimensions) only need:
  - `nrn` - Nullplatform Resource Name
  - `np_api_key` - API key
  - Module-specific variables (service_path, environments, etc.)
- **They do NOT need**: cluster_name, vpc_id, zone_ids, etc.

---

### 4. Do not add inline comments inside module blocks

Module blocks must be clean, without inline comments. Only use separation comments BEFORE the module block.

**Correct**: Separation comment BEFORE the block, no comments inside.
```hcl
# =============================================================================
# Module name
# =============================================================================
module "name" {
  source = "..."

  variable_1 = var.x
  variable_2 = local.y
}
```

---

### 5. NEVER hardcode values - always use variables

Every value in a module must come from a variable or local. DO NOT assign values directly (strings, lists). Declare the variable in `variables.tf`.

### 6. Consistent variable structure

For each variable used in main.tf:
1. Declare it in `variables.tf` with type, description, and default (if applicable)
2. If shared between layers, verify it's in `common.tfvars`

---

### 7. tags_selectors is a common variable

The `tags_selectors` variable is used in `nullplatform/` (service_definition) and `nullplatform-bindings/` (scope_definition_agent_association). Since they must always have the **same values**, this variable should be in `common.tfvars`.

---

### 8. Use OpenTofu (tofu) instead of Terraform

IaC commands must use `tofu` instead of `terraform`.

---

### 9. Layer-specific variables go in their terraform.tfvars, not in common.tfvars

- Only put in `common.tfvars` variables that are used in **more than one** layer:
  - `common.tfvars`: np_api_key, nrn, tags_selectors
  - `nullplatform/terraform.tfvars`: service_path, environments, service_spec_name, etc.

---

### 10. Provider API key vs `np` CLI may have different permissions

- Verify that the API key has the necessary roles for **all** module operations
- **Debug**: Test manually: `NP_API_KEY="<key>" np nrn patch --nrn "<nrn>" --body '{}'`

### 15. Show variable summary before tofu apply

**MANDATORY** before running `tofu apply`: show a table with ALL variables used by modules, indicating:

| Variable | Module(s) | Source | Value |
|----------|-----------|--------|-------|
| `var_name` | module that uses it | `common.tfvars` / `terraform.tfvars` / `default in variables.tf` | current value |

- Include variables with default values (not just those from tfvars)
- Mark sensitive variables as `(sensitive)`
- Indicate if any variable is declared but not used in any module
- Wait for user confirmation before running the apply

---

### 16. Module reused with different configurations

When the same module is used multiple times with different configurations (e.g., `scope_definition` for Containers and for Scheduled Tasks), it's necessary to pass variables with defaults that differ from the original value. This is an exception to the general rule of omitting variables with defaults.

**Example**: The `scope_definition` module has `service_path = "k8s"` by default. The first instance uses the default, but the second needs `service_path = "scheduled_task"`, `service_spec_name = "Scheduled Tasks"`, `action_spec_names = [...]`, etc.

**Rule**: When reusing a module, explicitly pass all variables that distinguish one instance from another, even if they have defaults in the module.

---

### 17. Scope definitions must have explicit descriptions

Each scope definition must include `service_spec_name` and `service_spec_description` with clear values. If the module has reasonable defaults for the first instance (e.g., `"Containers"` / `"Docker containers on pods"`), they can be used. For additional instances, always pass explicit values via variables.

**Rule**: Verify that each scope definition has a name and description defined, either by module default or by explicit variable.
