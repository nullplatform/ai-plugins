# Infrastructure Layer Generator

Generator for the `main.tf` of the **infrastructure/** layer in Nullplatform with OpenTofu.

> **IMPORTANT**: Do not assume values or configurations. When in doubt, divergence, or ambiguity, **always ask the user** before generating code.

> **MANDATORY VALIDATION**: Before generating or modifying any code, checklist:
> 1. Did I read the downloaded module's `variables.tf` AND `outputs.tf`? (Technical rule #7)
> 2. Did I include ONLY required variables? (no default = include, has default = include ONLY if the default needs to change)
> 3. Am I inferring any value that I should ask the user about?
> 4. Does `dns_type` match the chosen networking schema? (dns_type rule)
> 5. Am I using the module instead of creating resources directly?
> 6. Are the module blocks clean, without inline comments?
> 7. Are schema-dependent variables for Istio set with the exact required values from resources-by-cloud.md (not as placeholders)?
> 8. Am I passing variables with their existing default value? If so, remove them — only pass variables that override the default.

> **EXECUTION INSTRUCTIONS**:
> 1. Follow the question flow using `AskUserQuestion` - DO NOT assume configurations
> 2. Run `tofu init -backend=false` to download modules
> 3. **MANDATORY BEFORE GENERATING**: Read the `variables.tf` of EACH module from `.terraform/modules/` (downloaded version). DO NOT generate main.tf without having read ALL variables.tf first. Variable names in this document may be outdated; the source of truth is always the downloaded module.
> 4. Generate main.tf using ONLY variable names read from downloaded modules. Include ONLY variables without default + variables whose default needs to change. Do NOT dump all variables from variables.tf into the module block.
> 5. Run `tofu validate` to verify
> 6. If validate fails, fix BEFORE considering it done
> 7. **DO NOT make changes** to already generated code without explicit user confirmation

```

├── provider.tf
```

## Question flow

### Step 1: Base configuration
1. **Cloud provider**: aws | azure | gcp
2. **Output directory**: where to generate (default: `./`)

### Step 2: Backend
- **Create new backend or use existing?**
  - **Create new** -> Generates `backend/` folder with `infrastructure/{cloud}/backend` module
  - **Use existing** -> Requests existing remote storage data

### Step 3: Base infrastructure (create or existing)

For each component, asks whether to create or use existing:

- **VPC/VNet**: Create -> `infrastructure/{cloud}/vpc` module | Existing -> requests `vpc_id`, uses data sources
- **Cluster (EKS/AKS/GKE)**: Create -> `infrastructure/{cloud}/eks|aks|gke` module | Existing -> requests `cluster_name`, uses data sources for endpoint, CA, OIDC
- **DNS Zones**: Create -> `infrastructure/{cloud}/route53|cloud-dns|azure-dns` module | Existing -> requests `public_zone_id`, `private_zone_id`

**Nested subzones**: If the domain follows the pattern `{child}.{parent}.nullapps.io` (e.g., `staging.acme.nullapps.io`), two public DNS zone modules are needed: one for the parent zone (`acme.nullapps.io`) and one for the child zone (`staging.acme.nullapps.io`). The parent zone acts as delegation hub. Include both in `main.tf` (e.g., `module.dns_parent` and `module.dns`). See step 5 in SKILL.md for the full delegation flow.

**When a resource is existing**: declare variable + data source in `data.tf`. Modules that depend on the resource use the data source instead of `module.X.output`.

### Step 4: Networking Schema

The ALB Controller is **always required**. Ask for schema:

**Istio Schema (Recommended):**
- ALB Controller, Istio, Cert Manager + IAM, External DNS + IAM, Prometheus, Agent + Agent IAM, Base

**ACM + Ingress Schema (Alternative):**
- ALB Controller, ACM, Ingress Controller, External DNS + IAM, Prometheus, Agent + Agent IAM, Base

## Kubernetes/Helm Providers

Provider configuration depends on whether the cluster is CREATED or EXISTING:
- **Created cluster**: use EKS/AKS/GKE module outputs (read `outputs.tf` from downloaded module)
- **Existing cluster**: use data sources (`aws_eks_cluster`, `aws_eks_cluster_auth`)

> **CRITICAL**: Do NOT guess or invent output names for provider configuration. You MUST read the actual `outputs.tf` of the downloaded cluster module (in `.terraform/modules/`) BEFORE writing `provider.tf`. For example, AWS EKS outputs are `eks_cluster_endpoint`, `eks_cluster_ca`, `eks_cluster_name` — NOT `cluster_endpoint`, `cluster_certificate_authority`, `cluster_name`. See [aws.md](aws.md#new-cluster-created-by-tofu) for the exact provider blocks per cloud.

**Helm v3.x**: Use `kubernetes = { ... }` syntax with `=` (object), NOT `kubernetes { }` (block). Reference: Helm Provider v3 Upgrade Guide.

**AWS Profile**: If `aws_profile` is supported, use conditional in exec args to add `--profile` only if not null.

## Providers per layer

Each layer has its own `provider.tf` with ONLY the providers it needs. Do NOT copy the infrastructure providers into other layers.

| Layer | Providers needed | Why |
|-------|-----------------|-----|
| `infrastructure/` | `aws` (or cloud), `kubernetes`, `helm`, `nullplatform` | Manages cloud resources, K8s components via Helm, and nullplatform agent/base |
| `nullplatform/` | `nullplatform` only | Only creates nullplatform resources (scope_definition, dimensions, service_definition) |
| `nullplatform-bindings/` | `aws` (or cloud), `nullplatform` | The `asset/ecr` module needs AWS, the rest are nullplatform resources |

The `kubernetes` and `helm` providers are ONLY needed in `infrastructure/` — do not add them to `nullplatform/` or `nullplatform-bindings/`.

## Generation rules

### Do not assume, always ask

1. **NEVER assume variable values** - Always ask
2. **When in doubt, ask** - It's better to ask too much than generate incorrect code
3. **If a module requires unmentioned variables, ask**
4. **If you're not sure which components to include, ask**

### Technical rules

1. **Only include code for the selected cloud**
2. **Module version**: run `git fetch --tags && git tag --sort=-v:refname | head -1` in the `tofu-modules` repo to get the latest release. The `git fetch --tags` is mandatory to avoid using outdated tags from the local clone.
3. **Required providers**:
   - AWS: `hashicorp/aws ~> 6.0`, `hashicorp/kubernetes ~> 2.0`, `hashicorp/helm ~> 3.0`
   - Azure: `hashicorp/azurerm ~> 4.0`, `hashicorp/kubernetes ~> 2.0`, `hashicorp/helm ~> 3.0`
   - GCP: `hashicorp/google ~> 5.0`, `hashicorp/kubernetes ~> 2.0`, `hashicorp/helm ~> 3.0`
   - All: `nullplatform/nullplatform` - Check the latest version before generating:
     ```bash
     curl -s "https://registry.terraform.io/v1/providers/nullplatform/nullplatform/versions" | jq -r '[.versions[].version] | sort_by(split(".") | map(tonumber)) | last'
     ```
     Use `~> 0.0.X` with the obtained version. DO NOT hardcode a fixed version.
4. **Sensitive variables** marked correctly
5. **Data sources** for existing resources (not redundant variables)
6. **Format with `terraform fmt`** after generating
7. **Read READMEs, variables.tf AND outputs.tf** of each module from `.terraform/modules/` before generating
8. **NEVER invent or guess output names** - Read the actual `outputs.tf` of each downloaded module. Common mistake: using `cluster_endpoint` when the real output is `eks_cluster_endpoint`. Wrong output names cause `tofu validate` to fail on every first run.
9. **NEVER transform outputs between modules** - Pass outputs as-is (without `replace`, `regex`, `split`, etc.). If in doubt about the format a variable expects, read the module's internal code (main.tf, iam.tf, locals.tf) to see how it's used, don't infer by variable name

## Correct module structure in infrastructure/

**IMPORTANT**: Every module source MUST follow this exact pattern. Do not invent or modify the source format:
```hcl
source = "git::https://github.com/nullplatform/tofu-modules.git//{path}?ref={version}"
```
The `{path}` is the comment next to each module below. The `{version}` is a git tag (e.g., `v1.52.0`). See [tofu-modules-patterns.md](tofu-modules-patterns.md#module-source-git-ref) for details.

```hcl
# Cloud infrastructure
module "vpc" { }           # infrastructure/{cloud}/vpc
module "eks" { }           # infrastructure/{cloud}/eks (or aks/gke)
module "dns" { }           # infrastructure/{cloud}/route53 (or cloud-dns/azure-dns)

# ALB Controller
module "alb_controller" { }                    # infrastructure/{cloud}/aws_load_balancer_controller

# K8s components (depend on module.alb_controller)
module "istio" { }           # infrastructure/commons/istio
module "prometheus" { }      # infrastructure/commons/prometheus

# IAM (no depends_on, pure AWS resources)
module "alb_controller_iam" { }  # infrastructure/{cloud}/iam/aws_load_balancer_controller_iam (creates IAM role + K8s service account)
module "external_dns_iam" { }    # infrastructure/{cloud}/iam/external_dns
module "cert_manager_iam" { }    # infrastructure/{cloud}/iam/cert_manager
module "agent_iam" { }           # infrastructure/{cloud}/iam/agent

# Components using IAM
module "external_dns_public" { }   # infrastructure/commons/external_dns (type = "public")
module "external_dns_private" { }  # infrastructure/commons/external_dns (type = "private", create_namespace = false)
module "cert_manager" { }          # infrastructure/commons/cert_manager

# Security + Nullplatform (GO IN INFRASTRUCTURE, NOT IN BINDINGS)
module "security" { }  # infrastructure/{cloud}/security
module "agent_api_key" { }  # nullplatform/api_key (type = "agent")
module "base" { }            # nullplatform/base (receives security_group_ids from security)
module "agent" { }           # nullplatform/agent

# Optional (without Istio)
module "acm" { }     # infrastructure/{cloud}/acm
module "ingress" { } # infrastructure/{cloud}/ingress
```

**IMPORTANT**: For each module, read its README and `variables.tf` from `.terraform/modules/` after `tofu init -backend=false`. DO NOT copy variables from this document - they may be outdated.

**Reference**: The source of truth for modules and variables is `nullplatform/tofu-modules` (branch `main`). Always read downloaded modules in `.terraform/modules/`.

## Dependency chain (depends_on)

Modules deploying Helm charts depend on `module.alb_controller`.

### Dependency graph

```
vpc
 ├──► eks ──► alb_controller_iam ──► alb_controller
 ├──► dns                                  │
 │                    ┌────────────────────┼──────────────────┐
 │                    ▼                    ▼                  ▼
 │              istio              external_dns_public   cert_manager
 │                                      │
 │                                external_dns_private
 │
 ├──► security
 │         │
 │         ▼
 │       base (receives security_group_ids from security)
 │         │
 │         ▼
 │       agent
 │
 └──► (schema without Istio)
       dns ──► acm ──► ingress
```

| `alb_controller_iam` | `[module.eks]` |

| `security` | `[module.eks]` |
| `base` | `[module.alb_controller]` |
| `agent` | `[module.base]` |

| `acm` (without Istio) | `[module.dns]` |
| `ingress` (without Istio) | `[module.alb_controller, module.acm]` |

### Security module

The `infrastructure/{cloud}/security` module creates security groups for gateways. Its outputs are passed to the `base` module. Read security's `outputs.tf` and base's `variables.tf` from downloaded modules to know exact output and input names.

### Why does external_dns_private depend on external_dns_public?

Both attempt to create the `external-dns` namespace. To avoid conflict:
- `external_dns_public`: creates the namespace (default `create_namespace = true`)
- `external_dns_private`: uses `create_namespace = false` and depends on the public one

### Why NOT use depends_on on IAM modules or asset/ecr?

- IAM modules are pure AWS resources (not Helm charts), they don't depend on the webhook
- The `nullplatform/asset/ecr` module (in nullplatform-bindings/) has local provider configurations; `depends_on` generates errors

## dns_type rule based on networking schema

### 30. Agent dns_type depends on networking schema

The `dns_type` variable of the agent module **has NO default value**, and its value must match the schema:

| Schema | dns_type | Additional agent variables |
|--------|----------|---------------------------|
| **Istio** | `external_dns` | `service_template`, `initial_ingress_path`, `blue_green_ingress_path` |
| **ACM + Ingress** | `route53` | None |

- Do not mix schemas

### 33. Schema-dependent variables for Istio are MANDATORY

When the user selects Istio, the agent HTTPRoute template variables MUST be set with the exact values defined in [resources-by-cloud.md — Agent HTTPRoute Templates](resources-by-cloud.md#agent-httproute-templates-istio-schema--mandatory). These variables MUST NOT be left empty or as placeholders — set them with their required values directly in `variables.tf` and `terraform.tfvars`.

| Cloud | agent_image_tag |
|-------|-----------------|
| **AWS** | `"aws"` |
| **Azure/GCP/OCI** | `"latest"` |

Determined automatically, NOT asked to the user.

## How to read module variables

When reading a downloaded module's `variables.tf`:

1. **Variables without `default`**: always include them, they are required
2. **Variables with `default` that have a `validation` block**: read the `error_message` to understand in which context they are required. If the context applies (e.g., `cloud_provider = "aws"`), include them
3. **Variables with `default` without `validation`**: include only if the default needs to be changed

**IMPORTANT**: Do not skip step 2. Variables with conditional `validation` are the most common cause of errors in `tofu plan`. Always review all `validation` blocks of each module before generating code.

## Wiring between modules: api_key

### 43. Use api_key module to generate API keys at runtime

DO NOT pass `var.np_api_key` directly to runtime modules (agent, base). Use the `nullplatform/api_key` module:

- In infrastructure/: create `module "agent_api_key"` with `type = "agent"`, then pass `module.agent_api_key.api_key` to `module.base` and `module.agent`
- `var.np_api_key` is still used to authenticate the nullplatform provider and modules that only need auth

Available types: `agent`, `scope_notification`, `service_notification`, `custom`.

## Lessons learned (gotchas)

### 1. Duplicate variables in tfvars
Shared variables (`aws_region`, `np_api_key`, etc.) go only in `common.tfvars`, not in each `terraform.tfvars.example`.

### 2. Module outputs can change between versions
Always verify `outputs.tf` of the downloaded module. DO NOT assume output names.

### 3. Undeclared variables
Each layer must declare ALL variables it uses in `variables.tf`, including those from common.tfvars.

### 4. Backend bucket in remote state
Use `backend_bucket` variable from common.tfvars, DO NOT hardcode.

### 5. Optional AWS Profile
Add optional `aws_profile` variable (default = null). Use conditional in kubernetes/helm provider exec args to add `--profile` only if not null.

### 6. Backend module
Use the `infrastructure/{cloud}/backend` module directly. DO NOT create S3 resources manually. Read the downloaded module to know actual variables/outputs.

### 7. Module variables change between versions
Read `variables.tf` of the downloaded module to know current variables. DO NOT assume variables from one version exist in another.

### 8. Gateways namespace
The `gateways` namespace must exist before applying cert_manager. Apply order is: infrastructure -> nullplatform -> nullplatform-bindings.

### 9. Provider stubs don't require authentication
For Azure stub: use `resource_provider_registrations = "none"` (not `skip_provider_registration`).

### 10. Use existing IAM modules
NEVER create `aws_iam_policy` or `aws_iam_role` directly. Use modules from `infrastructure/{cloud}/iam/`.

### 11. NEVER hardcode values
Every value in a module must come from a variable. `dns_type = var.dns_type`, NOT `dns_type = "route53"`.

### 12. Consistent variable structure
For each variable in main.tf: declare it in `variables.tf` + add it to `terraform.tfvars.example` + if shared, to `common.tfvars.example`.

### 13. Module location
`base`, `agent`, and `agent_api_key` go in `infrastructure/`, NOT in `nullplatform-bindings/`.

### 14. tags_selectors is a common variable
Same `tags_selectors` variable in infrastructure/ and nullplatform-bindings/. Define it once in `common.tfvars`. DO NOT create `agent_tags_selectors`.

### 15. Use OpenTofu
Commands: `tofu`, not `terraform`.

### 16. S3 backend with profile
The `backend "s3"` block must include `profile` if using an AWS profile.

### 17. Dependencies and External DNS
See "Dependency chain (depends_on)" section above. Key points:
- Helm modules depend on `module.alb_controller`
- External DNS: `type` and `zone_type` are separate variables - always pass both
- Private External DNS: `create_namespace = false`, depends on the public one

### 18. Layer-specific variables
Only put in `common.tfvars` variables used in more than one layer. Infrastructure variables go in `infrastructure/terraform.tfvars`.

### 19. Backend module - verify version
Read variables.tf and outputs.tf of the downloaded module. The module may not have variables in certain versions.

### 20. Destroy order
Reverse of creation: nullplatform-bindings -> nullplatform -> infrastructure -> backend.

### 21. Orphaned Helm releases
If destroy fails due to deleted cluster, remove K8s resources from state with `tofu state rm` and re-run destroy.

### 22. Credentials expire during long destroys
If `AuthFailure`, renew credentials and re-run `tofu destroy`.

### 23. S3 Object Lock COMPLIANCE
Cannot be bypassed. Wait for retention to expire.

### 24. API key permissions
Verify the API key has roles for all module operations (provider + `np` CLI).

### 25. Read downloaded modules
Run `tofu init -backend=false` FIRST. Read from `.terraform/modules/`, NEVER from the repo root.

### 26. Confirm before modifying
Do not modify already generated code without explicit user confirmation.

### 27. Verify reverts
After reverting, `grep -r` to verify no residual references remain.

### 28. Validate immediately
After generating: `tofu init -backend=false && tofu validate`. Fix before considering it done.

### 29. Mandatory outputs for downstream layer consumption

The `infrastructure/` layer MUST export these outputs in `outputs.tf` so that `nullplatform-bindings/` can consume them via remote state:

| Required output | Description | Consumer in bindings |
|----------------|-------------|---------------------|
| `cluster_name` | K8s cluster name | asset_repository |
| `domain_name` | Application domain | cloud_provider |
| `public_zone_id` | Public DNS zone ID | cloud_provider |
| `private_zone_id` | Private DNS zone ID | cloud_provider |

Output values depend on the modules used. Read `outputs.tf` of each downloaded module to know real names. If a resource is added that other layers need, always export it as output.

### 30. Show variable summary before tofu apply

**MANDATORY** before running `tofu apply`: show a table with ALL variables used by modules, indicating:

| Variable | Module(s) | Source | Value |
|----------|-----------|--------|-------|
| `var_name` | module that uses it | `common.tfvars` / `terraform.tfvars` / `default in variables.tf` | current value |

- Include variables with default values (not just those from tfvars)
- Mark sensitive variables as `(sensitive)`
- Indicate if any variable is declared but not used in any module
- Wait for user confirmation before running the apply
