# Categories

Las categorías agrupan action items y definen defaults de comportamiento (config + units).

## Schema

Ver `model.md` para el schema completo. Resumen:
- `nrn`, `name`, `slug` (autogenerated), `description`
- `parent_id` (jerarquía max 2 niveles)
- `color`, `icon` (UI)
- `unit_name`, `unit_symbol` (unidad de la `value` field de los action items)
- `config` (defaults heredados por los action items)

## `config` semantics

```json
{
  "max_deferral_days": null,
  "max_deferral_count": null
}
```

| Field | Effect |
|-------|--------|
| `max_deferral_days` | Máximo de días hacia adelante que un defer puede fijar `deferred_until` (contados desde hoy). Se valida en cada defer de forma independiente — **no** es acumulativo entre defers |
| `max_deferral_count` | Máximo de veces que un item puede diferirse. Se valida en cada defer |

Estos defaults se heredan al crear el action item. Cada action item puede sobreescribir su propio `config` para casos especiales.

> Que un `defer` / `reject` / `resolve` requiera aprobación lo decide la política de aprobaciones de la plataforma; `max_deferral_days` y `max_deferral_count` se validan en cada `defer`.

## Categorías recomendadas

| Category | Slug | Description | Recommended config |
|----------|------|-------------|-------------------|
| Security Vulnerability | `security-vulnerability` | CVEs, security issues | `max_deferral_days: 90` |
| Cost Optimization | `cost-optimization` | Rightsizing, unused resources | (defaults) |
| Performance Optimization | `performance-optimization` | Slow queries, bottlenecks | (defaults) |
| Technical Debt | `technical-debt` | Refactoring, version upgrades | (defaults) |
| Compliance | `compliance` | Regulatory requirements | (defaults) |
| Deprecation | `deprecation` | Deprecated APIs/libs | (defaults) |

## Units recomendadas

| Unit name | Symbol | Used for |
|-----------|--------|----------|
| Dollars | `$` | Cost savings |
| Risk Score | `R` | Vulnerability severity (0–100) |
| Milliseconds | `ms` | Latency improvements |
| Hours | `h` | Time saved |
| Percentage | `%` | Efficiency improvements |

## Pattern: ensureCategory

Antes de que un agente cree action items, debe asegurarse de que la categoría existe. El script `ensure_category.sh` hace search-or-create:

```bash
CATEGORY_ID=$(${CLAUDE_PLUGIN_ROOT}/skills/np-governance-action-items/scripts/ensure_category.sh \
  --nrn "organization=1" \
  --slug "security-vulnerability" \
  --name "Security Vulnerability" \
  --description "CVEs and security issues detected by automated scanners" \
  --color "#DC2626" \
  --icon "shield" \
  --unit-name "Risk Score" \
  --unit-symbol "R" \
  --config '{"max_deferral_days": 90}')

echo "Category: $CATEGORY_ID"
```

Si la categoría existe (match por `name` + `nrn`, que es la clave de unicidad de la API), el script imprime el id existente. Si no existe, la crea y retorna el id nuevo. Es **idempotente** y seguro de correr en cada inicio del agente.

## Jerarquía padre-hijo

```
Cost Optimization (parent, nrn=organization=1)
  ├── EC2 Rightsizing (child, nrn=organization=1:account=2)
  └── RDS Idle Instances (child, nrn=organization=1:account=2)
```

Reglas:
- Max 2 niveles de profundidad
- El parent puede estar en un NRN ancestor del child
- El child no puede tener children (validado al crear)
- Borrar un parent falla si tiene children

## NRN scope y herencia visual

Las categorías son visibles en su NRN propio Y en sus descendientes. Ej: una categoría con `nrn=organization=1` aparece en queries que filtren por `nrn=organization=1:account=5:application=20`.

Esto se maneja server-side en `GET /governance/action_item_category?nrn=...`.
